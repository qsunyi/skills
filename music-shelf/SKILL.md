# MusicShelf

MusicShelf 用来把混乱的本地音乐目录整理成可长期维护的正式音乐库。

它的核心目标不是“把文件搬一遍”，而是建立一套可反复运行、可增量更新、可回刷规范化的音乐库整理流程。

## 适用场景

当用户需要以下任一事情时，应该优先使用此技能：

- 整理本地音乐库
- 把下载目录、备份目录、散乱资源整理进正式音乐库
- 归档无损音乐与 MP3 音乐
- 按歌手/专辑重建目录结构
- 给专辑补年份并统一命名
- 把繁体目录、乱码目录、下载站尾巴规范化
- 压平多余嵌套目录
- 去重、查重、保留更优版本
- 建立或更新本地 SQLite 索引
- 对既有目标库做“回刷规范化”
- 只处理新增资源，做增量整理

不适用于：

- 云音乐歌单管理
- 流媒体服务管理
- 音频转码工作流（除非用户明确提出）
- 单纯播放器功能

## 执行前必须确认的输入

在真正开始整理前，必须先确认以下输入：

1. 源头目录
2. 目标无损库目录
3. 目标 MP3 库目录
4. 本次是全量整理还是增量整理

如果用户没有明确提供这些信息：

- 先暂停执行
- 不要自行猜测路径
- 不要擅自把某个现有目录当作源头或目标
- 先要求用户补全后再继续

推荐先输出一段执行上下文确认，例如：

```text
本次执行将使用：
- 源头目录：
- 目标无损库：
- 目标 MP3 库：
- 处理模式：全量 / 增量
```

这些输入确认之后，才进入正式整理。

## 总体模型

始终把流程分成两个环境来看：

### 1. 源头目录 → 目标库

默认规则：

- `copy`，不 `move`
- 不自动删除源头曲目目录
- 不批量清空源头目录

这个阶段的目的，是把资源安全地纳入正式库，而不是改写源头结构。

### 2. 目标库内部整理

默认规则：

- `move`，不 `copy`
- 允许改名
- 允许改归类
- 先做文件名清洗，再决定是否压平、并回或保留多碟层
- 允许压平多余嵌套目录
- 允许做目标库回刷规范化

这个阶段的目的，是把正式库内部变得稳定、整洁、可长期维护。

> 这两个阶段必须严格区分。源头目录强调安全保留，目标库内部强调高效调整。

## 固定库名

正式库名固定为：

```text
Music_无损
Music_mp3
```

- `Music_无损`：正式无损库
- `Music_mp3`：正式 MP3 库

## 推荐目录结构

### 无损库

对于需要长期维护的艺人目录，建议将“单曲 / 合作曲 / 多版本”作为同级目录管理，而不是继续嵌在单曲下：

```text
歌手_A/
  单曲/
  合作曲/
  多版本/
  1991-专辑名[格式]/
  2004-专辑名[格式]/
```

- `单曲/`：普通独唱单曲
- `合作曲/`：合唱、对唱、联名演绎、多人参与的曲目
- `多版本/`：同一首歌的不同形态，如完整版、清唱版、演奏版、口白版、演唱版等

如果现有库里已经把这三类放在“单曲/”下面，允许先做目录提升，再逐步整理内容。


```text
Music_无损/
  00_待判定/
  歌手_A/
    1991-专辑名[FLAC]/
    专辑名[APE+CUE]/
    单曲/
  群星/
  原声大碟/
  90_附件素材/
  99_日志/
```

### MP3 库

```text
Music_mp3/
  歌手_A/
  群星/
  00_待判定/
```

## 核心整理原则

### 1. 艺人优先

第一层目录优先使用：

- 歌手
- 乐队
- 演奏者
- 作曲家
- 明确的项目名

如果同一位艺人存在多种写法、繁简混写、旧译名、常见笔误或变体目录，必须并回同一个规范艺人根目录。

例如：

- `周华健` 与 `周华键` → 统一到 `周华健`
- `苏慧伦` 与 `苏慧倫` → 统一到 `苏慧伦`
- 其他明显指向同一艺人的拼写变体，也应合并到一个规范根目录，而不是并列长期存在。

对于海外艺人，规范根目录建议增加地区前缀：

- 英文 / 欧美歌手：`ENG-`
- 日文 / 日本歌手：`JPN-`
- 韩文 / 韩国歌手：`KOR-`

注意：不要仅凭目录名称是 ASCII/英文就直接判定为欧美艺人；例如 `Beyond` 这类港乐/华语体系目录，不应因为名字是英文就草率加 `ENG-` 前缀。必须结合专辑、曲目语言、日志、封面和整体归属一起判断。

例如：

- `Adele` → `ENG-Adele`
- 日本歌手目录 → `JPN-歌手名`
- 韩国歌手目录 → `KOR-歌手名`

如果历史库里已经存在未加前缀的海外艺人目录，应在目标库回刷阶段统一并入带前缀的规范目录。

但是不要只看名字就判断是不是海外艺人：像 `Beyond` 这类目录，名字本身不能直接说明归属，必须结合里面的专辑名、歌曲名、语言、封面、日志和曲目说明一起判断；如果整套目录明显还是华语/港澳体系，就不要因为名字是 ASCII 就草率打 `ENG-` 前缀。

不要把这些内容误判成歌手名：

- 专辑名
- 情绪标题
- 系列标题
- 合辑名
- 精选名
- 盒装目录名
- DVD 容器目录名
- 顶层的“艺人 + 专辑/系列/版本说明”外壳目录

例如：

- `凤凰传奇 - 十年经典珍藏集(24K金碟) WAV`
- `孙露 - 试音女王`

这类目录不是最终根目录，应在目标库内部整理阶段并回规范艺人根目录，再把其中的专辑目录或单曲内容归到正确位置。

如果无法确认单一歌手，优先归入：

- `群星/`
- `原声大碟/`
- `00_待判定/`

特殊顶层类目处理口径：

- `男歌手/`、`女歌手/` 这类按性别分箱的容器目录，不应在正式目标库长期保留；应继续压平并回各艺人目录
- `欧美/` 不应作为长期兜底总类；优先拆回具体艺人目录，海外艺人再按 `ENG-`、`JPN-`、`KOR-` 等规范前缀归位
- `Others/` 只用于极少数暂时无法稳定并入 `群星/`、`原声大碟/`、`00_待判定/特殊格式` 的特殊资源；如果已经能明确归类，就不应继续留在 `Others/`

### 2. 专辑优先于单曲

当资源不是一个干净的专辑目录时，优先顺序是：

1. 先查本地专辑索引
2. 命中已有专辑时，直接并回该专辑目录
3. 没命中再补查专辑信息与年份
4. 只有在无法可靠确认所属专辑时，才进入 `单曲/`

这条原则非常重要：

> 只要能合理归专辑，就不要先放进单曲目录。

### 2.1 单曲归专辑必须联网查证

当单曲文件名不足以无歧义判断所属专辑时，必须先查网络权威曲目资料再决定：

- 优先查 MusicBrainz、Discogs、唱片厂牌/发行页、权威曲目表
- 只有网络证据与本地目录线索一致时，才允许并回专辑
- 不能靠文件名、记忆或主观推测直接归位
- 查不到可靠证据时，先进入 `00_待判定/`

推荐来源优先级：

1. 本地 `cue` / `log` / `txt` / 标签 metadata
2. MusicBrainz
3. Discogs
4. 唱片厂牌、发行方、官方/半官方曲目页
5. 可信的音乐资料站、维基型曲目页

使用规则：

- 低优先级来源不能直接推翻高优先级来源
- 多个来源冲突时，先保留到 `00_待判定/`
- 如果网络来源只能证明“这首歌存在”，但不能证明“属于当前库中的哪一版专辑”，仍然不能自动并回

### 2.2 独唱 / 合唱 / 版本冲突先待判定

当出现以下情况时，不要直接覆盖或并回：

- 独唱版 vs 合唱版
- 粤语版 vs 国语版
- Live vs Studio
- Remix / 伴奏 / 特别版
- 同名但来源不明的重复曲目

这类条目优先进入 `00_待判定/`，待后续核对后再决定保留、并回或去重。

### 2.3 证据等级

仅在满足以下任一证据时，才视为“可无歧义自动并回专辑”：

1. 目录内已有同专辑轨目、cue/log、封面等强一致线索
2. 网络权威曲目资料明确命中，且与本地文件标题一致
3. 已存在的正式专辑目录中能直接验证该曲目的归属

若证据不足，只做标记，不自动归位。

### 2.4 单曲归位决策树

处理单曲时，按下面顺序判断：

1. **先查本地证据**
   - 看 `cue` / `log` / `txt` / 标签 metadata / 同目录说明文件
   - 如果本地已经能无歧义确认归属，直接并回专辑

2. **再查目标库现有专辑**
   - 看目标库中是否已有明显对应的正式专辑目录
   - 看该专辑是否已出现同批次、同主题、同版本的其他曲目
   - 如果能和现有专辑形成强一致，直接并回

3. **本地不足时查网络**
   - 按“本地 metadata → MusicBrainz → Discogs → 厂牌/发行页 → 可信资料站”的顺序查
   - 只有在网络证据与本地标题、版本信息不冲突时，才可并回

4. **遇到版本冲突先暂停**
   - 独唱 / 合唱
   - 国语 / 粤语
   - Live / Studio
   - Remix / 特别版 / 伴奏
   - 同名但无法确认来源的重复
   - 以上情况一律先入 `00_待判定/`

5. **仍不能确认就不要猜**
   - 保留在 `单曲/` 或移入 `00_待判定/`
   - 不要为了“看起来整理完了”而强行归位

### 3. 年份前置优先，目标库回刷兜底

凡是判定为专辑目录，落入目标库前必须先判断是否能可靠拿到年份。

规则如下：

- **拿得到年份**：必须命名为 `年份-专辑名[格式]`
- **拿不到年份，但专辑名明确**：仍然落入正式专辑目录，命名为 `专辑名[格式]`
- **后续回刷阶段如果补齐年份**：再统一改名为 `年份-专辑名[格式]`

目录命名优先形式：

```text
歌手名/年份-专辑名[格式]
```

兜底形式：

```text
歌手名/专辑名[格式]
```

## 中文规范化

在匹配、归类、命名之前，先统一做中文规范化。这个阶段的目标不是“美观”，而是尽量把同一内容归到同一个可比较的名字上，避免因为编码、字体、半角全角或下载站尾巴把同一专辑拆成两份。

中文规范化不仅适用于目录名、专辑名和艺人名，也适用于歌曲文件名以及歌曲基础标签（如 `title`、`artist`、`album`、`album_artist`）。只要本地证据足够、且不会引入歧义，就应在首次整理时一次完成这些基础信息的简体化与规范化，避免后续再开补丁式修复流程。

优先做这些清洗：

- 繁体转简体
- 清理明显的乱码字符、控制符、错码片段（mojibake）
- 去掉首尾空格与重复空格
- 统一半角 / 全角符号
- 统一括号、连接符、分隔符风格
- 去掉下载站尾巴、广告后缀、重复来源标记
- 尽量把轨道号统一为 `01-歌曲名.ext`

处理时尤其要注意：

> 繁体与简体差异、乱码错码、全角半角差异，都不应该导致同一歌手、同一专辑、同一首歌被拆成两份。

如果文件名含有明显的 mojibake / 错码字符，但能从同目录的封面、日志、曲目说明或其他文件名看出原名，先按这些线索做修复；如果仍然不确定，就把它放入 `00_待判定/`，不要猜。

对于合集、精选、原声、盒装、混装这类目录，若标题里出现了下载站来源尾巴、重复的父级前缀、或明显多余的品牌标记（例如把来源站名直接粘在专辑名后面），也应视为可清洗项；清洗时优先保留真正的专辑名 / 合集名 / 碟号信息，去掉会干扰归类的附加标记。
### 可无歧义自动执行

下面这些动作如果不需要额外判断，就可以优先自动做，尽量先把名字清洗干净，再进入归类和归位：

1. 只清理空格、标点、半角 / 全角和常见尾巴
2. 对完全一致的重复候选做去重或保留标记
3. 对明显只是繁体 / 简繁差异的命名先统一后再归位
4. 对无法确认原名或归属的内容保留到 `00_待判定/`

清洗完以后再判断要不要并回艺人根目录、要不要压平盒装层、要不要保留多碟层。
## 格式与库规则

### 无损优先入 `Music_无损`

例如：

- FLAC
- APE
- WAV
- DTS
- WV

### MP3 单独入 `Music_mp3`

- 不把 MP3 直接混进 `Music_无损`
- `Music_mp3` 内部也按歌手/群星/待判定来组织

### 同曲目双库并存

如果同一首歌同时有：

- 无损版本
- MP3 版本

则：

- 无损版本进入 `Music_无损`
- MP3 版本进入 `Music_mp3`
- 两边都保留，不互相替代

## 重复策略

### 同歌不同格式

在同一个库里优先级如下：

1. FLAC
2. APE
3. WAV
4. 其他无损
5. MP3

### 单曲 vs 专辑分轨

如果 `单曲/` 中某首歌与专辑分轨是同一内容：

- 默认保留专辑分轨版本
- 单曲目录中的重复项不再额外保留
- 如果后来确认专辑归属，应优先并回专辑目录

### 合辑 / 精选集

- 如果合辑里的曲目与原专辑中的版本完全相同，不要因为它又出现在合辑里就重复入库
- 如果是不同版本，例如 `Live`、`Remix`、`伴奏`、`特别版`，则可以保留

## 嵌套目录处理

目标库中如出现无意义嵌套目录，必须继续压平整理。遇到 `DVD/盒装/群星混装` 这类目录时，先做文件名清洗，再判断它是应该：

- 继续保留为一个有语义的多碟层
- 并回到艺人根目录
- 还是拆成多个艺人目录 / 合集目录

不要因为文件名里有繁体、乱码、全角符号或下载站尾巴，就直接跳过这一步；先把名字清洗到能比较、能判断的状态，再决定下一步。

典型坏结构：

- `歌手/盒装目录/盒装目录/专辑/`
- `歌手/DVD目录/DVD目录/专辑/`
- `Others/Others/内容/`
- `歌手/DVD23刘若英/DVD23刘若英/专辑/`
- `艺人 + 专辑/艺人 + 专辑/内容/`
- `艺人/DVD17艺人/艺人/专辑/`

其中，像 `凤凰传奇 - 十年经典珍藏集(24K金碟) WAV`、`孙露 - 试音女王` 这类顶层“艺人 + 专辑/系列/版本说明”壳目录，应先并回规范艺人根目录，再处理内部内容。对于 `艺人/DVD编号艺人/艺人/专辑` 这种结构，要继续压平为 `艺人/DVD编号艺人/专辑`；如果 `DVD编号艺人` 本身也只是纯容器编号，则继续压到 `艺人/专辑`。

### 合集 / 盒装 / 混装盘的处理顺序

处理 `群星/`、`原声大碟/`、`DVDxx艺人`、`盒装合集`、`多艺人混装盘`，以及其他本质上属于“合集 / 壳层 / 混装”的目录时，建议按这个顺序执行：

1. 先清洗目录名和文件名
2. 识别这层是不是纯容器层
3. 如果只是纯容器层，就继续往下看真实专辑层
4. 如果内容已经能明确归到单一艺人，就并回对应艺人根目录
5. 如果内容本来就是合辑 / 原声 / 群星精选，就继续留在合适的合集根目录，例如 `群星/` 或 `原声大碟/`
6. 如果一个目录已经是合理的合集集合，且内部没有混装艺人壳、重复容器层或明显误归档内容，就停止继续拆散，只进入命名一致性修复模式
7. 如果父级目录已经清楚表达了“这是合集 / 原声 / 分类总集”，则子目录标题里重复出现的父级前缀可视为冗余，清洗时可以去掉，但不要删除真正有区分度的合集名、碟号或版本信息
8. 如果存在 `CD1`、`CD2`、`Disc 1/2`、`HQCD`、`DVD ISO` 这类有语义层，就保留这些层，不要压掉
9. 如果仍然无法可靠判断归属，就放入 `00_待判定/`

压平原则：

- 如果父目录与子目录只是重复一层容器名，就把子目录内容上提一层
- 如果某层本身有信息价值，则保留

必须保留的语义层示例：

- `CD1`
- `CD2`
- `Disc 1/2`
- `DVD ISO`
- `HQCD`
- `Booklet`
- `扫描图`
- `Cover`

一句话：

> 保留有语义的层，去掉纯包裹、纯重复、纯容器式嵌套。

## 索引方案：默认 SQLite

不要用 JSONL 作为主索引方案。默认采用 SQLite。

### 索引位置

```text
Music_无损/.library_index/music_index.db
Music_mp3/.library_index/music_index.db
```

### 为什么选 SQLite

因为这个任务长期需要：

- 增量处理
- 专辑查重
- 歌曲查重
- 年份补查
- 目标库回刷规范化
- 频繁查询
- 双库联查

在这种场景里，SQLite 比 `JSONL + 内存索引` 更简单，也更稳定。

## SQLite 实现设计

### 数据库初始化

如果目标库根下不存在：

```text
.library_index/music_index.db
```

优先使用内置脚本：

```bash
python scripts/init_index.py "/path/to/Music_无损" "/path/to/Music_mp3"
```

不要临时手写一套和 skill 不一致的建库逻辑。优先复用 `scripts/init_index.py`，并参考 `references/sqlite-schema.md`。

该脚本会：

1. 创建 `.library_index/`
2. 创建 `music_index.db`

- 目标库里是否存在重复或命名不一致？

## 入库前歌曲标签与封面标准化

当源头目录中的资源准备进入目标库时，除了做目录归类和命名规范化，还应尽量一次完成歌曲级别的基础 metadata 整理，避免把“文件结构整理好了、但歌曲信息仍然残缺或繁简混杂”的问题留到后续补救。

### 1. 基础标签最小补全

对歌曲文件，优先读取现有标签；如果以下核心字段都已存在且非空：

- `title`
- `artist`
- `album`

则默认视为“已有歌曲信息”，跳过补写。

如果缺失上述任一核心字段，则只从本地可验证证据补齐可确定字段，优先级如下：

1. 文件已有标签
2. 文件名中的曲序与曲名
3. 所在专辑目录名
4. 上级歌手目录名
5. 同目录 `cue` / `log` / `txt` / 其他说明文件

可以补的最小字段是：

- `title`
- `artist`
- `album`

在证据明确时，也可补：

- `album_artist`
- `track`

不要主观猜测作词、作曲、流派、年代说明、版本说明等扩展字段；这些内容只有在本地证据明确且用户要求处理时才进入本轮流程。

### 2. 歌曲文件名与标签简体统一

在首次整理阶段，除目录名外，还应同步处理歌曲文件名和基础标签的简体化，避免出现“目录已规范，但歌曲文件名和标签仍是繁体或混用”的遗留状态。

默认规则：

- 对歌曲文件名做繁体转简体
- 对 `title`、`artist`、`album`、`album_artist` 做繁体转简体
- 如果标签已存在，则不因为字段已齐全而跳过简体统一
- 只统一语言形态，不凭空扩写内容

例如：

- `遠在咫尺` → `远在咫尺`
- `陳奕迅` → `陈奕迅`
- `斯德哥爾摩情人` → `斯德哥尔摩情人`

### 3. 专辑封面嵌入规则

如果专辑目录中存在明确封面文件，应在支持安全写入封面的格式上，默认把封面嵌入歌曲文件，而不是只保留外部图片。

默认规则：

- 优先使用 `cover.jpg`
- 若没有 `cover.jpg`，再从同目录图片中选择最明确的主封面
- 对支持嵌入封面的音频格式，默认写入专辑封面
- 若歌曲已存在封面，且本轮模式属于“已有就跳过”，则跳过覆盖
- 若格式或工具链不适合安全写入封面，则保留原状并说明原因
- 非歌曲图片文件默认只参与命名规范化，不修改图片内容本身

### 4. 目录内非歌曲文件名简体化

专辑目录中的非歌曲文件（如 `jpg`、`png`、`log`、`txt`、`cue`、`pdf`）虽然不一定直接参与播放，但它们常常是目录可读性、后续排障和本地证据判断的一部分，因此在首次整理时也应同步做文件名简体化。

默认规则：

- 只改文件名，不改文件内容
- 优先处理与专辑强相关的图片、日志和说明文件
- 如果繁体转简体后会与现有文件重名，则不要覆盖，先记录冲突

### 5. 执行口径

在 `music-shelf` 的默认工作流里，应把这部分理解为“入库前收口动作”：

- 目录归类前后都可以做，但必须在本轮正式落库结束前完成
- 目标是让资源第一次进入正式库时，就已经具备可用的基础标签、统一的简体命名，以及尽可能完整的封面信息
- `music-shelf-tagging` 这类补救型流程应只用于历史遗留目录回刷，而不应成为默认主流程的第二段

## 增量处理设计

后续源头目录如果新增了曲目、专辑或整批资源，默认按“增量整理”执行，而不是每次全量重扫。

### 三层模型

#### 1. 目录级候选发现
先找：

- 新增目录
- 最近发生变化的目录
- 未处理目录

只把这些列入本轮候选。

#### 2. 文件级去重判定
对候选目录里的文件做比对：

- 规范化名称
- 歌手
- 专辑
- 曲目
- 时长
- 格式

避免把已经入库的内容再处理一遍。

#### 3. 目标库局部规范化
只对本轮命中的目标目录做回刷：

- 补年份
- 改目录名
- 压平嵌套
- 修正繁简不一致

不要每次全库重命名。

### 推荐保存的增量状态

- 已处理目录清单
- 已入库专辑索引
- 已入库歌曲索引
- 最近一次执行时间
- 最近一次执行日志

### 增量顺序

1. 先发现新增/变更目录
2. 先处理新增专辑目录
3. 对目录名、歌曲文件名与基础标签做简体化和规范化
4. 对缺失基础标签的歌曲补齐 `title` / `artist` / `album`
5. 若目录中存在明确封面文件，对支持的格式嵌入封面
6. 再处理新增单曲、合辑、精选集
7. 最后只回刷本轮涉及到的目标目录

## 推荐执行顺序

1. 初始化或刷新 SQLite 索引
2. 扫描专辑型目录
3. 处理整轨 / 分轨专辑
4. 对名称做简体化和规范化
5. 年份前置判断
6. 优先归专辑
7. 无法归专辑的才放 `单曲/`
8. 处理 DTS
9. 处理 MP3，入 `Music_mp3`
10. 执行目标库回刷规范化：
    - 补年份
    - 压平嵌套
    - 修正格式标记
    - 修正繁简不一致

## 输出要求

真实执行时，应清楚说明：

- 哪些路径是源头目录
- 哪些路径是目标库
- 哪些内容是 `copy` 过去的
- 哪些内容是在目标库内部 `move` 的
- 哪些内容进入了 `单曲/`
- 哪些内容进入了 `00_待判定/`
- 哪些内容被判为重复候选
- 哪些内容还需要人工复核

## 合集壳目录上提规则

当某位艺人的根目录下同时存在：

- 已经在根目录落位的一批正式专辑 / 版本目录
- 一个额外的“全集 / 合集”壳目录，里面又包着同艺人的大量专辑目录

应优先把它识别为“合集壳目录上提”场景，而不是直接把整包目录长期保留。

推荐顺序：

1. 先清洗合集壳内子目录名
2. 去掉无歧义的错误前导符号，例如年份前多余的 `-`
3. 生成“规范化比较名”后，与艺人根目录现有目录做冲突比较
4. **只有在无冲突时**，才把子目录从合集壳中上提到艺人根目录
5. 如果上提目标与根目录已有目录在标题层面构成同专辑不同版本、同专辑不同格式或疑似重复，则保留在合集壳内，进入人工复核
6. 上提完成后，再复扫一轮艺人根目录，确认合集壳是否只剩冲突项、附件项或特殊版本项

这里的“冲突”不等于“可删除重复”。

例如同名专辑如果根目录已存在：

- `1989-SALUTE[WAV]`
- `1989-SALUTE[XRCD]`

而合集壳内还有：

- `1989- SALUTE`

这种情况应视为**版本冲突待判定**，不要自动删，也不要自动并。

附件 / 特殊项处理口径：

- `BIG COVER`
- `OTHER COVER`
- `BONUS CD`
- 特殊母带版、金碟版、XRCD2、扫描图包

这类目录只有在位置明显错误时才调整；如果只是“属于该艺人的附件或特殊版本集合”，可先上提或原位保留，但不要因为它不属于标准专辑名就直接删除。

## 零剩余收尾规则

一次整理任务在 skill 视角下，只有在“本轮可无歧义执行的动作已经做完”之后，才算真正收尾，而不是“做了一部分典型动作就停止”。

精修时必须完成的三件事：

1. **把能自动清洗的名字清洗干净**：先处理空格、标点、半角 / 全角、繁简和明显错码，再进入归类判断
2. **把合理的合集留下来**：已经是合理群星合集、原声集、精选集或多碟结构的，不要过度拆散；只对纯容器层和明显串档内容继续处理
3. **把零剩余复查做完**：再扫一轮，直到本轮还能自动做的动作清零；剩下的都明确记成人工判断或 `00_待判定/`

收尾时至少再做一轮复查：

1. 复查当前目录或当前命中的目标目录里，是否还残留可继续自动做的清洗项
2. 复查是否还存在纯容器层、重复容器层、明显可并回的艺人目录或群星混装壳
3. 复查是否还有完全一致的重复候选尚未标记或处理
4. 只有在剩余项都已经变成“需要人工判断”或“应放入 `00_待判定/`”之后，才报告这一轮完成

如果还有剩余工作，不要因为已经完成了主要动作就提前停下；应继续处理到本轮可无歧义动作清零，再把少量例外单独列出来。

## 护栏

- 不臆造歌手名
- 不把 MP3 混进 `Music_无损`
- 不自动删除源头曲目目录
- 不批量清空源头目录
- 不压平有语义的层级
- 根目录下的说明/备注 `.txt` 只要有内容，不能默认当垃圾删除；先判断它属于专辑备注、歌手总说明还是库级日志，再决定移动、改名或保留
- 不因为年份暂缺就阻止专辑正式落库
- 不跳过繁体转简体规范化
- 不把本地机器路径写死进规则

## 成功标准

MusicShelf 处理后的结果应当具备：

- 艺人优先
- 专辑优先于单曲
- 年份已知时带年份前缀
- 年份未知但专辑明确时仍能正式落库
- 能处理复杂嵌套目录
- 能长期增量维护
- `Music_无损` 与 `Music_mp3` 双库清晰分离
- 每个库都有独立 SQLite 索引
- 双库之间可联查重复与版本关系


## 重复文件处理补充

- 对于同一目录内出现的 `(2)` 变体，通常可先视为重复候选。
- 但在删除前，最好先做内容或文件大小比对；若能确认一致，再保留优版、删除重复项。

## 乱码修复补充

- 分轨文件名出现乱码时，优先检查本地 `cue`、`log`、`txt`、标签 metadata，再决定是否需要外部曲目表。
- 如果本地信息不足，再查外部可信曲目单，按曲序恢复标准曲名。
- 重命名时优先统一为 `01-歌名.ext` 这类稳定格式。
- 不要直接凭肉眼猜乱码文件名。
- 如果整轨镜像目录与分轨目录并存，不要直接合并；先确认它们是否只是不同封装版本，再分别保留或处理。


## 繁体字与乱码处理补充

- 乱码必须修；分轨文件名乱码时，优先按本地 `cue` / `log` / `txt` / metadata / 外部可信曲目单恢复。
- 繁体字不一概强制转简体。
- 歌名 / 曲目名如果能确认标准简体写法，可统一为简体。
- 专辑名 / 发行名 / 历史标题如果更像原版保留项，优先保留原样。
- 无法确认时，先进入待判定，不要冒然改写。
